> Принцип документа: де ТЗ лишає вибір — тут зроблено вибір із обґрунтуванням.

## 0. Головна теза архітектури

**MELOS — це не застосунок із музичним движком усередині. Це музично-теоретичний движок, у який вбудований застосунок.**

80% цінності і 90% ризику зосереджені в одній речі: **чистій, headless, повністю тестованій бібліотеці музичної теорії та аналізу на TypeScript**. UI, аудіо, нотація, Tauri-оболонка — це замінні шари над нею. Тому весь проєкт будується навколо `core/`, а не навколо екранів.

Наслідок для планування: спершу — домен, потім — модулі. Модуль M2 (Mode Explorer) без `core/theory` — це порожня вітрина; з ним — він майже безкоштовний.

## 2. Структура монорепо

```
F:\MELOS\
├─ docs/                      # цей план, ADR-и, нотатки з джерел
├─ packages/
│  ├─ core-theory/            # ЧИСТИЙ TS. Без UI, без аудіо, без I/O.
│  │   ├─ pitch, interval, scale, mode, chord, key
│  │   ├─ contour/            # Morris/Friedmann c-space, типи контуру
│  │   ├─ rhythm/             # метр, гіпермет, syncopation index
│  │   └─ cadence/            # розпізнавання PAC/IAC/HC/Deceptive/Plagal/Phrygian
│  ├─ core-analysis/          # алгоритми поверх theory: gap-fill, earworm score,
│  │   │                      #   golden-section climax, n-gram predictability
│  │   └─ (інтерфейс до Python-sidecar — з'являється на v0.3)
│  ├─ core-pedagogy/          # Finding → Explanation registry + bibliography refs
│  ├─ data/                   # JSON+Zod: modes, cadences, progressions,
│  │                          #   rhythmic-cells, canonical-examples, bibliography
│  ├─ audio/                  # Tone.js: sampler, scheduler, playback
│  ├─ notation/               # VexFlow (інтерактив) + LilyPond export adapter
│  ├─ srs/                    # FSRS для Ear Training (чистий TS, тестований)
│  └─ ui/                     # React-компоненти, дизайн-система (dark, gallery-core)
├─ apps/
│  ├─ web/                    # Vite+React shell (v0.1+). Запускається у браузері.
│  └─ desktop/                # Tauri-обгортка (додається на v0.3)
├─ sidecar/                   # Python: music21, тренування n-gram, обробка корпусів
│   └─ (білд-тайм скрипти + рантайм RPC-сервер, з v0.3)
└─ corpora/                   # сирі датасети + офлайн-пайплайни анотації (gitignore великих)
```

Чому пакети, а не одна тека `src/`: межі залежностей стають фізичними. `core-theory` за визначенням не може імпортувати React — це гарантує його чистоту й тестованість.

## 3. Шарова модель залежностей

```
        ┌─────────────────────────────────────────┐
        │  apps/web (UI shell)  →  apps/desktop     │   ← замінний
        ├─────────────────────────────────────────┤
        │  ui  │  audio  │  notation  │  srs        │   ← платформні шари
        ├─────────────────────────────────────────┤
        │  core-pedagogy   (Finding → Explanation)  │   ← педагогіка
        ├─────────────────────────────────────────┤
        │  core-analysis   (метрики, скоринг)       │   ← алгоритми
        ├─────────────────────────────────────────┤
        │  core-theory     (атоми домену)           │   ← фундамент
        ├─────────────────────────────────────────┤
        │  data            (лади, каденції, корпус) │   ← істина
        └─────────────────────────────────────────┘
```

Правило: стрілки лише вниз. `core-theory` не знає про `analysis`; `analysis` не знає про `pedagogy`; ніщо в `core-*` не знає про React чи Tone.js. Це робить весь movзок придатним для запуску в CLI/тестах/Node без браузера.

## 4. Доменна модель (`core-theory`) — серце системи

Це найважливіша і найризикованіша частина. Її треба спроєктувати першою і покрити тестами на 100%.

### 4.1 Базові типи
- **Pitch**: представлення `{ step, alter, octave }` + MIDI-номер + spelled-name. Критично: розрізняти `D#` і `Eb` (enharmonic spelling) — без цього неможливі правильні інтервали й голосоведіння. Tonal.js дає це з коробки; беремо його за фундамент і добудовуємо.
- **Interval**: напрямлений, з якістю (м.2/в.2/…/тритон/…/октава) і семітоновою величиною. Кожен інтервал несе `affect`-тег із Level 1 ТЗ (м.2=напруга, ч.5=порожнеча, …) — це дані, прив'язані до типу.
- **Scale / Mode**: інтервальна формула від тоніки + метадані (див. §5). Операції: `transpose`, `degreeOf(pitch)`, `isCharacteristicDegree`, `nearestScaleTone`.
- **Chord**: набір ступенів + функція (T/S/D за Riemann) у контексті ключа.
- **Key / TonalContext**: tonic + mode; дає функційну інтерпретацію акордів і ступенів.

### 4.2 Похідні структури
- **Melody**: послідовність `{ pitch, onset, duration }` (внутрішньо — у тіках/quarter-notes, не в секундах; темп — окремо). Це канонічна форма, у яку конвертуються MIDI/MusicXML/намальований контур/проспіваний звук.
- **Contour**: редукція мелодії до напрямків (Morris c-space). Класифікатор у тип (Ascending/Descending/Arch/Inverted-arch/Plateau/Undulating — Level 4/Savage).
- **RhythmGrid**: метрична сітка з гіперметром; основа M5 і Level 5.

### 4.3 Чому власні типи, а не «просто Tonal.js»
Tonal.js покриває pitch/interval/scale/chord-арифметику — беремо. Але він не знає про контур, каденції, гіперметр, earworm-фічі, gap-fill, золотий перетин кульмінації. Це все — наш `core-analysis` поверх Tonal-фундаменту. Tonal — це калькулятор; MELOS — це теорія.

## 5. Дані як істина (`data`) — схема ладу

Mode Explorer (M2, перший модуль v0.1) повністю керується даними. Схема одного ладу (Zod-валідована):

```ts
ModeDefinition {
  id, names: {uk, en}, aliases[],
  formula: number[],              // семітони від тоніки, напр. Dorian [0,2,3,5,7,9,10]
  parent?: { scale, degree },     // похідність (Dorian = 2-й ступінь Ionian)
  characteristicDegrees: number[],// які ступені дають «колір» (Dorian → ♮6)
  typicalCadences: CadenceRef[],
  typicalHarmony: ProgressionRef[],
  canonicalExamples: ExampleRef[],// 5–10 прикладів із посиланнями на slow-listening
  context: { emotional, geographic, historical }: {uk,en},
  family: 'church'|'melodic-minor-modes'|'symmetric'|'messiaen'|
          'ukrainian'|'balkan'|'raga'|'maqam'|'japanese'|'pentatonic'|'bebop'|'synthetic'
}
```

Та сама схема покриває весь Level 6: церковні, лади мел./гарм. мінору, октатоніку/цілотонну, 7 ладів Месіана, **українські (гуцульський, укр.-дорійський, лемківські обертональні)**, балканські, раги, макамо (з прапорцем мікротональності), японські, пентатоніки, bebop, synthetic. Макам/рага позначаються `microtonal: true` — рендер і аудіо мають із цим рахуватися (окрема нота нижче).

Аналогічні схеми: `CadenceDefinition`, `ProgressionDefinition` (з полем `whyItWorks` — для M7/M8), `RhythmicCell` (clave/tresillo/dotted/gallop/son/rumba…), `BibliographyEntry` (для M10, експорт у BibTeX/Zotero), `CanonicalMelody` (для M3 Hit DNA — з повним «генетичним паспортом»).

**Ключове:** усі модулі-вітрини (M2/M3/M5/M7/M8/M10) — це різні представлення одних і тих самих даних. Один раз наповнюємо — багато разів показуємо.

## 6. Шар аналізу (`core-analysis`) — алгоритми з джерелами

Кожен алгоритм прив'язаний до конкретного джерела з ТЗ §1. Це і є «прозорість»: метрика → джерело.

| Алгоритм | Джерело (ТЗ §1/§10) | Складність | Реалізація |
|---|---|---|---|
| Класифікація контуру | Morris/Friedmann; Savage 2015 | низька | детермінований TS |
| Syncopation index | Longuet-Higgins & Lee 1984 | низька | детермінований TS |
| Groove «Goldilocks» прогноз | Vuust et al. 2014 | середня | крива поверх syncopation index |
| Gap-fill / правило компенсації | Narmour I-R; Meyer | низька | детермінований TS |
| Гравітація вступних тонів (7→1,4→3,6→5) | Level 1 | низька | детермінований TS |
| Розпізнавання каденцій | Caplin; ТЗ Level 3 | середня | патерн-матч на акорд+сопрано |
| Кульмінація / золотий перетин (0.618) | Bartók; ТЗ Level 3 | низька | геометрія позиції |
| Earworm potential score | Jakubowski et al. 2017 | висока | зважена сума фіч (контур-arch, темп, малі ритм-фігури, спадні в.2 у кульмінації) |
| Predictability score (n-gram) | anti-banality; Berlyne inverted-U | висока | таблиця ймовірностей із §8 (офлайн-тренована) |
| Cliché penalty | ТЗ §4 | середня | словник заїжджених фігур + матч |

**Вихід кожного алгоритму — не число, а `Finding`**: `{ kind, severity, params, location, citationIds[] }`. Це контракт між аналізом і педагогікою.

### Anti-banality engine (ТЗ §4) — окремий критичний контур
Не модуль UI, а наскрізний сервіс, що споживає `Finding[]` і додає свої: cliché-penalty, predictability (n-gram), novelty/familiarity-баланс (Berlyne), genre-fingerprint-contamination. Видає три рівні «свіжості» альтернатив (⭐/⭐⭐/⭐⭐⭐). Тренувальні моделі (TheSession, Essen, Wikifonia, Bach chorales) готуються в `sidecar/` офлайн → компактні таблиці в `data/`.

## 7. Педагогічний шар (`core-pedagogy`) — як виконується «кожна порада з джерелом»

```
Finding (з analysis)  ──►  ExplanationRegistry[kind]  ──►  Explanation
                                                            { text:{uk,en},
                                                              citations: BibliographyEntry[],
                                                              algorithmLink,        // «подивитись алгоритм» (ТЗ §7-НЕ)
                                                              freshnessAlternatives? }
```

- Реєстр — це **дані** (шаблони + ID цитат), не switch у коді. Додати нову техніку = додати запис.
- LLM (локальна Qwen/DeepSeek) опційно перефразовує `text` у природніший коментар, **отримуючи вже готовий Finding** — він не вирішує, лише озвучує. Якщо LLM недоступний — показуємо шаблон. Жодна порада без `citations[]` не рендериться (інваріант, перевіряється в тестах).
- Це напряму реалізує ТЗ §7 «Що додаток НЕ робить»: немає «AI магічно зробив», є «техніка [X] з контексту [Y] → [Z], бо [причина: cite]».

## 8. Платформні шари

- **audio**: Tone.js. Семпли інструментів (не лише синт — для канонічних прикладів). Планувальник на тіках мелодії. Окремий виклик: **мікротональність** макам/раг — Tone.js дозволяє detune, але рендер UI і нотація мають показувати коми; на v0.1 макам/рага можуть бути «огляд» (як і каже ТЗ — «огляд»), повна мікротональність — пізніший трек.
- **notation**: VexFlow для інтерактиву (M1 piano-roll + нотація поруч), LilyPond-адаптер для статичного експорту (PDF/висока якість). LilyPond — зовнішній бінар → реально доступний лише в desktop (Tauri, v0.3+); у web-режимі експорт у MusicXML/MIDI, а LilyPond-PDF — фіча десктопу.
- **srs**: FSRS (новіший за SM-2, ТЗ допускає обидва — беремо FSRS). Чистий TS, прив'язаний до курикулуму: тренує саме ті елементи, з якими користувач працює в Composer's Lab (ТЗ M9), а не абстрактний тренажер.
- **ui**: dark-first, gallery-core, progressive disclosure, без gamification, шорткати на все (ТЗ §5). Дизайн-система як окремий пакет, щоб усі 10 модулів виглядали однаково.

## 10. v0.1 у деталях — що будувати першим

Конкретний обсяг першого релізу (офлайн, локально):

**Mode Explorer (M2):**
1. `core-theory`: Pitch/Interval/Scale/Mode + операції (transpose, degreeOf, characteristic degrees, nearestScaleTone).
2. `data`: Zod-схема `ModeDefinition` + наповнення ~15–20 ладами для старту (7 церковних + пентатоніки + гуцульський + укр.-дорійський + мел./гарм. мінор — те, що найближче користувачеві-ірландцю: Dorian/Mixolydian, і українському матеріалу).
3. `audio`: програвання гами/прикладу в будь-якій транспозиції.
4. UI: бібліотека ладів; для кожного — гама, характеристичні ступені, типові каденції, приклади, контекст; **«Try in this mode»** (миттєва транспозиція) і comparison-view (одна канва в 5 ладах).

**Ear Training (M9):**
1. `srs`: FSRS-движок (картки, планувальник повторень).
2. Типи вправ v0.1: розпізнавання інтервалів і ладів на слух + інтонування інтервалу голосом (YIN-pitch → порівняння з еталоном).
3. Прив'язка карток до ладів, які користувач відкриває в Mode Explorer.

**Інфраструктура під це:** монорепо (pnpm), Vite+React (web), Vitest для `core-*`, дизайн-система-заготовка (dark). Tauri — НЕ чіпаємо.

## 11. Ризики й відкриті питання

1. **Наповнення даних — не код, а праця музикознавця.** 500+ паспортів (M3), 5–10 канонічних прикладів на кожен лад (Level 6), бібліографія — це місяці курування, не програмування. Архітектура (D6) це визнає й робить наповнення інкрементним, але обсяг лишається. Користувач (ТЗ §9) уже це позначив як дослідницькі задачі.
2. **Ліцензії корпусів** (ТЗ §9): TheSession (CC), Essen, Wikifonia (статус неоднозначний), MAESTRO — перевірити терми ДО тренування anti-banality. Bach chorales з music21 — безпечні.
3. **Мікротональність макам/раг.** Чесна позиція: v0.1–v0.x — «огляд» (як і пише ТЗ), повна мікротональна нотація+аудіо — окремий пізній трек, бо чіпає і `notation`, і `audio`, і UI.
4. **Earworm/predictability метрики — це наука, не факт.** Jakubowski-модель тренована на захід. ТЗ §9 уже вимагає окрему укр. метрику — закладено в D4 (дві таблиці). Потрібен evaluation-pipeline (blind-тести з композиторами) до того, як довіряти скорам у UI.
5. **LLM-пояснення — degradation.** Інваріант: застосунок повністю функціональний без LLM (шаблони). LLM — це поліровка тексту, не залежність.
6. **Назва** (MELOS/KANTOR/ARIA) — продуктове рішення за вами; у коді використовується нейтральний `melos` як package-scope, перейменування дешеве.

## 12. Що НЕ робимо (фіксація меж з ТЗ §7)

- Не генеруємо пісню одним промптом.
- Жодної поради без `citations[]` — інваріант, перевірений тестом у `core-pedagogy`.
- LLM не приймає музичних рішень (отримує готовий `Finding`).
- Низький confidence показуємо чесно (`Finding.severity`/`confidence` рендериться в UI).
- Не gamification (без стріків/конфетті).
